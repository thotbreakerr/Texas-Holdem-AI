# Pre-training review — engine + MCCFR + Deep CFR (2026-06-05)

Read-only review (no code changed). Goal: can the user start training MCCFR (Path A)
or Deep CFR (Path B) safely. Method: project readiness gates run as primary evidence,
plus a 9-dimension static fan-out (engine, 2× engine↔CFR-state parity, 2× CFR algorithm,
2× trainer, shared utils, runnability), each finding adversarially verified against the
*declared* MCCFR+ / Deep CFR Plus variant. 16 findings confirmed, 9 rejected as
false-positives (correct-for-variant), 1 resolved to a config caveat.

## Bottom line

**Both paths are GO to start training.** All readiness gates pass, both CFR tree-games
faithfully mirror the real engine (the worst-case "train on a different game than you
play" risk did **not** materialize), and there are no won't-start blockers. The confirmed
issues all degrade *model quality* over a long run rather than crashing it — but a few are
worth fixing in the few minutes before kicking off an overnight run, because they silently
weaken the result and you won't see them until eval.

Triage order: **silent-corruption > won't-start > degraded-quality > cosmetic.**

**If you want to pick one to start tonight:** Path B's *core CFR learning loop is more
correct* — it has the linear iteration weighting and does **not** double-count opponent
reach. Path A has a genuine bias in its core regret update (A1 below). Path B's confirmed
issues are more peripheral (history-encoder skew, missing NaN guard, sizing granularity,
and a config gotcha) — real, but mostly "add a guard / set a flag" rather than core-math
fixes. So: **Path B is the cleaner core algorithm to launch as-is; Path A is worth the
~few-line A1 fix first.**

## Path A (MCCFR) — GO, two real algorithm issues

### A1. 🔴 HIGH — regret update double-counts opponent reach under external sampling
`bots/cfr_bot.py:1846-1854`

`_cfr_recurse` samples exactly one opponent action on-policy (line 1820) with **no**
importance correction, then multiplies the hero's instantaneous regret by
`cf_reach` = the product of opponents' reach probabilities (the sampled path
probability `q_-i`). In external-sampling MCCFR the opponent reach is *already*
carried by the sampling frequency, so multiplying by it again makes
`E[regret] ∝ π_-i²` instead of `π_-i`. The explicit `π_-i` factor is only correct
when opponent actions are **enumerated** (vanilla CFR) — here they are sampled.

- **Effect:** trains without crashing, but the average policy best-responds to a
  squared/sharpened opponent-reach distribution, starving rarely-reached
  defensive/exploit lines. Only visible via exploitability/eval after a long run.
- **Why HIGH, not blocker:** each re-solve inits `reach={s:1.0}`, so the *immediate*
  decision node (root seat-to-act) gets `cf_reach=1` and is unbiased; the bias lands on
  *deeper* hero nodes. A given info set is sometimes root and sometimes deep across
  traversals, so the persistent table is a *diluted* mix of biased/unbiased updates —
  still biased, still worth fixing, but not catastrophic.
- **Note:** Path B does *not* do this (its regret target is the raw `action_value − ev`),
  which is corroborating evidence this is a Path-A defect, not intended.
- **Fix direction:** drop the `cf_reach` multiplier on the regret update (keep the
  `self_reach` weight on `strategy_sum`, which is correct). A few lines.
- This is the single most worthwhile thing to fix before a real Path A run.

### A2. 🟡 MEDIUM — declared CFR+ linear-iteration weighting is absent
`bots/cfr_bot.py:1853-1856` (+ `_run_iterations` 1562-1573)

The declared variant is "MCCFR+ with linear iteration weighting + positive-regret
matching." `get_strategy` does floor at read time (positive-regret matching ✓), but
**no iteration index is threaded anywhere** — both `regret_sum` and `strategy_sum`
accumulate with weight 1. No `×t` weighting (Path B *does* have it: `weight = max(1, iteration)`
at `deep_cfr_bot.py:1961`). Also it's RM, not true RM+ (the accumulator isn't floored
at 0 each step — minor).

- **Effect:** still converges to Nash, just slower / weaker at a fixed training budget.
  Not wrong, just under-delivering the planned ~10–30% convergence boost.

### Path A parity — clean apart from one known, bounded reconstruction gap
`_GameState` faithfully mirrors `_betting_round`: legal-action construction, short-all-in
non-reopening, bet-vs-raise labeling, side-pot settlement, fresh-street action order all
match method-by-method. Depth-limit leaf and street-relabel-on-short-deck are
correct-for-variant.

One caveat (🟢 LOW): the auto-verifier dismissed "big-blind inferred from `pot×2/3`" as
correct-for-variant, but that is actually **FIX_REPORT Known Limitation #2** —
`_build_game_state_from_view` reconstructs `big_blind` *imperfectly*, which shifts the
reconstructed min-raise and therefore the history-token info-set key. It's a known,
*accepted* train/deploy reconstruction gap, **bounded by the passing
`sanity_cfr_sizing_parity` test** (it surfaces as the boundary-pinned bucket there). Not a
blocker and not "wrong," but it is a real bounded limitation, not a non-issue.

## Path B (Deep CFR) — GO, a cluster of train/serve skews + robustness gaps

### B1. 🟡 MEDIUM–HIGH — history-GRU is fed different features in training vs inference
`bots/deep_cfr_bot.py` + `core/action_history.py`

Two distinct train/serve skews on the action-history GRU input, both because the
tree builds richer `ActionEvent`s than the engine's `PlayerView.history` exposes at play:

- **all-in one-hot (`action_history.py:43`, idx 5 → channel 19):** the tree emits
  `action="all_in"` (`deep_cfr_bot.py:893-895`), but the real engine never emits all-in
  (it's a *flag* on a raise/bet spec), so `extract_history` reads the identical shove as
  `raise`/`bet` (channel 17/18). The net trains on channel-19 all-in prefixes and is
  queried on channel-17/18 at play.
- **call amount channel (`deep_cfr_bot.py:879/924` vs `engine.py:744`):** training stores
  `event_amount = cost` for a call; the engine forces call `amount=None` → `0` at
  inference. Every call's amount channel diverges.

- **Effect:** bounded (magnitude preserved; ~1–2 of 23 channels), but it specifically
  miscalibrates decisions downstream of a prior all-in — exactly the spots the recent
  all-in-canary work is trying to harden. Silent: smoke tests pass because both run the
  *tree* encoder; the gap only shows when the net is driven from real engine PlayerViews.
- **Fix direction:** make one encoder canonical — either map the abstract `all_in` label
  back to bet/raise in the tree constructor (match the engine), or derive all-in from the
  post-action all-in state in `extract_history`; and carry the call cost into engine
  history (or zero it in training) so both sides agree.

### B2. 🟡 MEDIUM — no NaN/inf guard before `optimizer.step()`
`training/train_deep_cfr.py:277-289`

`total_loss.backward()` → `clip_grad_norm_(max_norm=10)` → `optimizer.step()` with no
finiteness check. `clip_grad_norm_` does **not** protect against NaN: one NaN gradient
makes `total_norm` NaN → clip coef NaN → all grads NaN → `step()` writes NaN into every
parameter. Worse, the verifier confirmed a NaN net **slips past the canary** during the
0–150k warmup (NaN regret → `max(0,nan)=0` → uniform policy → all-in masked → canary PASS
→ promoted to `deep_cfr_v1.pt`). Realistic trigger: the unbounded value-head MSE target on
raw AIVAT chip-EV, or an MPS numerical edge case.

- **Effect:** conditional (needs an outlier), but if it fires on a long MPS run, every
  subsequent checkpoint is silently NaN-corrupted while the run reports normal losses and
  exits "complete." Cheap to guard (`torch.isfinite(total_loss)` skip-step).

### B3. 🟡 MEDIUM — postflop raise buckets partially collapse facing a bet
`bots/deep_cfr_bot.py:901` (`target_total = int(frac_map[label] * pot)`)

Computed as raw `frac × pot` then clamped to `[min_raise, max]`. Facing a bet, the
min-raise floor is high, so the smaller fraction buckets (bet_33/50/67) clamp up to the
same min-raise. cfr_bot was fixed for exactly this (`_raise_sizing_target` =
`current_bet + frac*(pot+to_call)`, `cfr_bot.py:485-502`); Deep CFR was not.

- **Verifier correction:** the collapse is **partial** (full only at pot-sized villain
  bets, ~6/30 scenarios) and crucially **self-consistent across train/eval** (training
  apply, inference `_abstract_to_concrete`, and the engine all use the same `frac*pot`
  mapping). So this is *not* a sim-vs-engine divergence — it's reduced postflop sizing
  granularity. Degraded-quality, not silent-corruption.

### B4. ⚙️ CONFIG GOTCHA — the default `--iterations 100000` produces an all-in-incapable model
`training/train_deep_cfr.py:46-48, 1037`

Schedule: `WARMUP=100k` (shadow), `DEPLOY=150k` (inference unmasks all-in),
`FULL_RELEASE=350k` (staged→full). But `--iterations` **defaults to 100k**. A
default run keeps all-in in shadow the entire time and saves a checkpoint
(`training_iteration=100k < 150k`) whose inference path **permanently masks all-in**.

- **Action:** run with `--iterations 1000000` exactly as TRAINING_PLAN.md step 7 says
  (≥350k minimum). The default is effectively a shadow-only smoke config.

### B5. 🟢 LOW / OPS — SIGINT exits 0; SIGTERM loses the checkpoint
`training/train_deep_cfr.py:770-774, 942-962, 1077-1082`

The SIGINT handler sets a flag and breaks (no re-raise), so the run returns
`status="complete"` and exits 0 even when interrupted — an orchestrator keying on exit
code would treat an under-trained model as finished. Manual Ctrl-C still saves a valid
checkpoint (fine for interactive use). SIGTERM is never trapped, so an OS sleep/kill loses
the in-flight checkpoint. (Not the canary path, which *does* abort nonzero — FIX_REPORT's
claim is about canary, not signals.)

### Path B parity — clean
`_DeepCFRGameState` mirrors the engine; `_build_search_game_state` reconstruction verified

present; the all-in curriculum masking (Key Change #2) is intentional and gated; leaf

## Shared / engine — clean

- **Engine** (`core/engine.py`): betting-round closure (incl. the lone-live-player fix,
  `570-575`), side pots (`calculate_side_pots`), and showdown settlement
  (`_showdown_and_settle` correctly intersects pot-eligibility with non-folded hands) are

  straight, ties, and odd-chip splits pass.
- **AIVAT** (`core/aivat.py`): `chip_ev` returns gross pot share; side-pot settlement is
  chip-conserving; the tournament fold-baseline redistributes the pot so fold-world and
  play-world share the same chip total. The `- added_cost` subtraction in both CFR
  recursions is **correct** given the gross-share leaf (not a double-count).
- 🟢 **LOW latent** — `PlayerView.min_raise/max_raise` use *increment* semantics while
  `legal_actions`/apply use *total-contribution* semantics (`engine.py:646-679, 797-804`).
  Both bots sidestep it by sizing from `legal_actions`, so no training impact today; it's a
  trap for any future consumer that trusts those fields.
