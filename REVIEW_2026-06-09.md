Data flow worth knowing: the engine never tells a bot the blind sizes, the
street-committed amounts, or "all-in" as an action type. Both CFR paths
therefore *reconstruct* blinds/contributions from `history` (`cfr_bot.py:184-381`),
and several findings below are consequences of that reconstruction boundary.

## Phase 2 — Findings, ranked

Legend: 🔴 critical (invalidates results or training), 🟠 important (silently
weakens an agent), 🟡 minor.

### 🔴 C1 — MLP preflop hand-strength feature differs between training and inference

`training/train_ml_bot.py:202-208` vs `bots/ml_bot.py:161-169`.

Training computes `hand_strength = eval_hand(hole, board) / EVAL_HAND_MAX`
unconditionally. Preflop, `eval_hand` returns the rank-sum heuristic (max ≈ 68,
see `engine.py:135-140`), so dividing by `EVAL_HAND_MAX` (134,217,740) makes the
feature ≈ 0 **for every preflop training sample** — AA and 72o identical.
Inference (`ml_bot.py:168-169`) correctly switches to `score / _PREFLOP_EVAL_MAX`,
producing values up to 1.0. The model is trained with a dead preflop feature,
then queried with a live one it has never seen in that range. The comment in the
trainer ("normalised the same way as MLBot._estimate_hand_strength()") is wrong.
Postflop (≥5 cards) the two sides agree.

This is exactly the class of bug the README claims was fixed ("ML bot feature
alignment") — the BB-position 0.5/0.3 mismatch was fixed, this one was not.

### 🔴 C2 — PPO reward plumbing: terminal bonus is a silent no-op, and per-hand rewards are duplicated across every decision

Three interacting defects in `bots/rl_bot.py` + `training/train_rl_bot_selfplay.py`:

1. **Terminal bonus never lands.** `record_reward` (`rl_bot.py:432-447`) tags
   steps from `self._hand_step_start` onward, then advances the marker. The
   trainer calls it once per hand (`train_rl_bot_selfplay.py:252-255`), so after
   the final hand the marker already equals `len(current_episode)`. The
   subsequent `rl_bot.record_reward(final_bonus)` (`:269-271`) iterates an
   **empty slice** — the +1.0/−0.5 win/loss bonus tags zero steps. The README's
   "terminal bonuses for wins/losses" does not actually reach the gradient.
   (`train_multi_deep_rl_bot.py` documents "no terminal bonus" and is consistent.)

2. **Per-hand reward counted once per decision.** Every step of a hand gets the
   full hand chip delta as its own per-step reward (`rl_bot.py:443-444`), and the
   GAE recursion (`rl_bot.py:521-531`) then *sums* per-step rewards. A hand with
   3 decisions injects ≈3× (discount-weighted) the actual chip delta into the
   returns, with earlier decisions inflated most (`R·(1+γλ+…)`). Hands with more
   decisions dominate the gradient for no principled reason. Standard fixes:
   reward only on the hand's last step, or divide the hand reward across steps.

3. **Reward scale is stack-relative, not stake-relative.** The per-hand reward
   divides by the *current* pre-hand stack (`train_rl_bot_selfplay.py:253-255`,
   `train_multi_deep_rl_bot.py:253`): winning 50 chips at a 10-chip stack is
   reward 5.0; the same 50 chips at 500 is 0.1. Combined with (2), short-stack
   hands swamp the batch. BB-normalisation (like Path B uses) would be stable.

GAE mechanics themselves are correct (bootstrap indexing, terminal V=0,
clipped-ratio PPO, exploration actions logged with their true policy log-prob).
The defects are all in what the algorithm is fed.

### 🔴 C3 — Silent fallbacks mean current tournament/eval results don't measure what they claim

`bots/__init__.py:75-82`: bare `cfr` checks whether the default profile exists
and, if missing, passes `profile_path=None` — deliberately routing **around**
CFRBot's own loud-fail guard (`cfr_bot.py:2061-2067`). The bot then plays
uniform-prior search but is scored as "cfr". Similarly `ml`/`rl` print a
one-line warning and play their fallback heuristic (`ml_bot.py:83-87`,
`rl_bot.py:114-121`).

In this checkout `models/` contains **none** of: `cfr_regret_deep_v2.pkl`,
`ml_model.pt`, `rl_model.pt`, `deep_cfr_v1.pt` (only `deep_cfr_large_*.pt`
artifacts exist). Consequences today:

- `run_tournament.py:41` default pool (`…,ml,rl,cfr,…`) and
  `run_tournament_stats.py:322-324` (`…,cfr,…`) silently run 3 heuristic
  stand-ins labeled as learned bots. Any win-rate table produced from these
  defaults misattributes results.
- Bare `deep_cfr` loud-fails (good), but its default points at a filename no
  trainer run has produced (`deep_cfr_v1.pt` vs actual `deep_cfr_large_*.pt`).
- `train_rl_bot_selfplay.py` saves `models/rl_model_run3.pt` while the factory
  loads `models/rl_model.pt` — a freshly trained RL bot is not picked up unless
  the user remembers `rl:models/rl_model_run3.pt`.

run_eval.py itself is safe (it always passes explicit `cfr:<path>` /
`deep_cfr:<path>`), and its Wilson-CI/verdict logic checks out.

### 🟠 I1 — MCCFR regret update double-counts opponent reach (re-verified; prior A1)

`bots/cfr_bot.py:1818-1825` samples exactly one opponent action on-policy
(external sampling — the sample frequency already carries π₋ᵢ), then
`:1845-1855` multiplies the regret by `cf_reach` = ∏ opponent reach again.
Expected update weight becomes ∝ π₋ᵢ² instead of π₋ᵢ. Root-level updates
(reach=1) are unbiased; all deeper hero nodes are biased toward
high-probability opponent lines. Fix is a few lines: drop `cf_reach` from the
regret increment (keep `self_reach` on `strategy_sum` — that part is correct).

### 🟠 I2 — Path A history tokens diverge between tree training and real play (new)

Inside `_GameState.apply_action`, tree actions append tokens by **bucket label**
(`cfr_bot.py:924-929`: bet_33→"S", bet_50→"Q", …). Real engine history is
tokenized by **realized ratio** `amount / pot_before`
(`core/action_history.py:151-176`). For opening bets these coincide by
construction, but for **raises** they cannot: a raise's engine amount is
`current_bet + frac·(pot + to_call)` (`cfr_bot.py:485-502`), so the realized
ratio lands 1-2 buckets higher (preflop raises almost always tokenize as
"P"/"A" regardless of bucket). Result: strategy mass learned for subtrees under
"…S" keys is looked up in real play under "…A" keys — the multi-action lines
Path A learns are partially orphaned at deploy time. Min-raise clamping causes
the same drift postflop (the spots `sanity_cfr_sizing_parity` pins as
"boundary-pinned").

### 🟠 I3 — Deep CFR history-GRU train/serve skew on all-in and call channels (re-verified; prior B1)

`deep_cfr_bot.py:893-895` emits `action="all_in"` ActionEvents in the tree;
`core/action_history.extract_history` (`:95-108`) can never produce `all_in`
from engine history (the engine has no such action type), and engine call
entries have `amount=None` → 0 while tree calls store the paid cost
(`deep_cfr_bot.py:871-879`). The net is trained on history channels that real
play never activates — precisely in post-all-in decision contexts the all-in
curriculum is trying to harden.

### 🟠 I4 — Deep CFR has no average-strategy mechanism (plays last-iterate)

Deep CFR (Brown et al. 2019) plays a *policy network* trained on a strategy
buffer (σₜ weighted by t); only the average strategy converges to Nash —
the last iterate can oscillate. This implementation has no strategy buffer and
no policy head: inference regret-matches the final advantage net
(`deep_cfr_bot.py:1499-1531`). The reservoir-with-linear-weights regret targets
(`train_deep_cfr.py:219-241`, `deep_cfr_bot.py:1956-1962`) do approximate
*time-averaged* regrets, which softens but does not remove the gap. Worth
knowing when eval results look unstable between adjacent checkpoints (compare
the long `warn_*` checkpoint sequence in `models/`).

### 🟠 I5 — No finiteness guard before `optimizer.step()` (re-verified; prior B2)

`train_deep_cfr.py:277-289`: one NaN loss → `clip_grad_norm_` propagates NaN to
every gradient → all parameters NaN, and subsequent canaries can still PASS
during warmup (NaN regrets → uniform policy → all-in masked). A 3-line
`torch.isfinite(total_loss)` skip-step is cheap insurance for 24-hour runs.

### 🟠 I6 — Deep CFR trains on a single state distribution: 6 players, 1000 chips, 5/10 blinds, equal stacks (new)

`train_deep_cfr.py:141-181` (`build_initial_state`) is the *only* source of
training states: always 6 seats, all stacks exactly 1000, blinds 5/10, hero
rotated by iteration. Stack asymmetry only arises within a single hand's tree.
But tournaments are decided heads-up, short-stacked, at escalated blinds —
states the value/regret nets have never sampled (`opp_mask` supports fewer
opponents, but the trained distribution never exercises it). This silently
caps the strongest agent exactly where titles are won. A curriculum that
randomises player count (2-6), stack distribution, and blind level would match
the eval distribution (`run_eval.py:42-44`: 500 chips, 5/10, ×1.5/50 — note
even the *starting* effective stack differs from training: 100BB vs 50BB).

### 🟠 I7 — PPO has no legality masking; gradients credit actions that were never executed (new)

`rl_bot.py:336-362` samples from the unmasked 6-way softmax; conversion then
silently substitutes a different action when the sampled one is illegal
(`_action_idx_to_action`/`_choose` fallback chains `:389-414`, plus engine-side
sanitization `engine.py:725-743`). The PPO update (`:556-574`) uses the
*sampled* index with the reward earned by the *executed* action — e.g. "check"
reinforced when a call actually happened, "fold" when a check did. Masking
illegal logits to −inf before softmax fixes both training and eval play.

### 🟠 I8 — Multi-opponent RL trainer's "CFR" opponent is an untrained, online-learning CFRBot (new)

`train_multi_deep_rl_bot.py:62,87-89`: profile `models/cfr_regret.pkl` doesn't
exist → `CFRBot(profile_path=None)` with `inference_mode=False`, which runs
`iterations=100` full `_cfr_recurse` traversals **per decision**
(`cfr_bot.py:1379-1383`) while learning from scratch. The advertised "CFR
opponent" is therefore (a) a drifting target, (b) mostly-uniform early, and
(c) a ~100× slowdown on every CFR decision of every episode.

### 🟠 I9 — `train_deep_cfr.py` default `--iterations 100000` yields an all-in-incapable model (re-verified; prior B4)

Defaults at `train_deep_cfr.py:46-48,1037`: deploy gate 150k > default 100k →
checkpoint saved with `training_iteration=100000` masks all-in at inference
permanently. Default invocation is a footgun; either raise the default or warn
when `iterations < all_in_deploy_iteration`.

### 🟠 I10 — Path A's inference subgame search effectively argmaxes single-sample noise (new)

`cfr_bot.py:1887-1894`: refined weight = `prior · exp(ev − max_ev)` with `ev`
in **chips** — a 20-chip EV gap is e²⁰, so the trained average strategy (the
prior) is irrelevant and the bot plays argmax over EVs estimated from *one*
sampled opponent action per node (`:1936-1938`) with no opponent cards
(`_build_game_state_from_view` gives opponents no holes → their info-keys get
bucket 0 → unseen → uniform, `:1516-1525`). Path B solved this exact problem
(temperature 20, ±2 clip, 0.25 blend, `deep_cfr_bot.py:1140-1142,1744-1762`);
Path A should adopt the same shaping or drop search and trust the table.

- **M9 — ML/RL feature scale is bound to `starting_chips=500` defaults**; the

- **M10 — ML memory features differ train vs inference** (part of C1's family

- **M11 — Fixed eval seating**: PATH_A always seat 0, PATH_B seat 1

## Ranked improvement roadmap (top 5)

**1. Fix the PPO training signal: reward plumbing + action masking (C2 + I7 + M5).**
The RL bot is the only agent whose core learning signal is structurally broken
in three compounding ways: the documented terminal bonus never reaches a
gradient, every hand's reward is multiplied by its decision count, and
gradients regularly credit actions that were never executed. Until these are
fixed, more episodes mostly amplify noise — which is consistent with the bot
needing curriculum hand-holding to beat a random player. Fix: tag the hand
reward once (last step, or split evenly), add the bonus to the final hand's
steps before `end_episode`, normalise rewards by big blinds instead of current
stack, and mask illegal logits to −inf before softmax (this also upgrades eval
play). Add the missing unit test for `record_reward`/`_ppo_update`. Effort:
~1-2 days including tests; no architecture changes.

**2. One shared feature/label builder for the supervised+RL family (C1 + M10 + M8 + M9).**
Extract a single `core/features.py` that both `MLBot._make_features`,
`RLBot._make_features`, and `PokerDataset` import, with the preflop
hand-strength branch and one VPIP/aggression definition, parameterised by
`starting_chips`. Add a parity sanity (`features(train_row) ==
features(inference_view)` for a recorded hand) mirroring
`sanity_cfr_sizing_parity`. This kills the current preflop-feature mismatch,
the memory-feature divergence, and the duplication that already produced one
shipped bug (BB 0.5/0.3), and prevents the next one. Effort: ~1 day.

**3. Path A core math: drop `cf_reach` from the regret update, add linear
weighting, and unify raise tokens (I1 + M3 + I2).** The regret fix is a
few lines and removes a real convergence bias; linear weighting (`×t`, as Path
B already does) is a documented-but-missing ~10-30% convergence speedup; the
raise-token fix (tokenize tree actions from their *realized* amount/pot ratio
instead of the bucket label — one line in `apply_action` reusing
`_canonical_tokenize` semantics) stops multi-street lines from being learned
under keys that live play never hits. Do these before the next long Path A
run; they change the table contents, so plan a fresh retrain. Effort: ~½ day
of code + a retrain.

**4. Make Deep CFR train on the game it will be evaluated on (I6 + I3 + I5 + I9).**
Randomise `build_initial_state` over player count (2-6), stack distribution
(e.g. log-uniform 10-200BB with asymmetry), and blind level so heads-up and
short-stack endgames are in-distribution; canonicalise the history encoding
(map the tree's `all_in` label back to bet/raise + zero call amounts, or derive
all-in in `extract_history`) so the GRU sees identical channels in training and
play; add the `torch.isfinite` skip-step guard; raise the default
`--iterations` (or refuse to save a "final" checkpoint below the deploy gate).
This is the highest-leverage *strength* work for the strongest agent: it fixes
the exact regime (short, heads-up, escalated blinds) where tournaments are
decided. Effort: ~2-4 days + retrain (the curriculum/canary machinery already
in place makes this lower-risk than it sounds). If appetite remains, adding a
strategy buffer + policy head (I4) is the principled next step.

**5. Loud-fail artifact policy + single source of truth for model paths (C3 + M9).**
Make `create_bot` fail (or require an explicit `--allow-fallback`) when a
learned bot's artifact is missing, instead of silently substituting a
heuristic; align the factory defaults with what the trainers actually write
(`rl_model_run3.pt`, `deep_cfr_large_*.pt`), or better, have trainers write a
`models/MANIFEST` the factory reads. Today the default tournament pools
quietly grade three fallback heuristics as "ml/rl/cfr" — every comparison
table produced that way is measuring the wrong thing. This is a small change
(~½ day) that protects the validity of every future eval.

Honorable mentions (not top-5 but cheap): tame Path A's inference search with
Path B's clip/temperature/blend recipe (I10, ~hours); fix the multi-RL
trainer's CFR opponent to load a real frozen profile in `inference_mode=True`
(I8, ~minutes once a v2 profile exists); collapse the six private equity loops
onto `core/equity.py` with a `valid`-count denominator (M1+M8, ~hours).
